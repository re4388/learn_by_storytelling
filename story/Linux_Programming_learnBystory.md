1. 虛擬記憶體佈局與分頁機制 (Virtual Memory Layout and Paging)

3. 程序與訊號處理的競態條件 (Race Conditions in Processes and Signals)

# 寓言故事一：紙莎草城的图书馆

## 故事

很久很久以前，有一座繁榮的紙莎草城，城中有一座聞名遐邇的龐大圖書館，名叫「永恆卷宗庫」。

這座圖書館驚人之處在於：它收藏的卷軸數量，远远超出圖書館本身的容納能力。館內陳列的架子只有一百格，卻號稱能容納十萬卷軸。原因是——绝大多数卷軸並不是真正放在架子上，而是抄寫在城外各處莊園的羊皮紙上，只有在需要的時候，才會被召回館中陳列。

國王派遣了一位聰慧的學者，名叫佩吉，來管理這座圖書館。佩吉首先為每一卷軸編製了一本目錄簿。這本目錄簿記載的不是卷軸的標題，而是每一卷軸被劃分成的每一張羊皮紙的存放位置。例如：卷軸甲的第7張羊皮紙來自北莊園的4號箱，第15張來自南莊園的2號箱，以此類推。這本目錄簿，就叫做「頁面索引簿」。

圖書館的規則如下：

- **調閱卷軸**：學者前來借閱卷軸時，管理員佩吉先查閱目錄簿，找出所有相關羊皮紙目前在館內架上的位置，然後將它們從城外各處召回，整齊擺放在架子上，學者就能閱讀了。
- **空間不足時**：若架子已滿，佩吉必須先選擇一些較久未被調閱的羊皮紙歸還原莊園，清出架子空間，再將新需要的羊皮紙放上來。這就是「頁面置換」。
- **卷軸的分類**：每卷軸的開頭是「稱謂區」，記載卷軸的名字與用途；中間是「正文區」，是實際內容；末端有「堆疊區」，用於書寫臨時筆記。這三個區域分別對應圖書館的不同展示區——稱謂區的羊皮紙質地最堅固，通常不需頻繁更換；而堆疊區的羊皮紙則經常流轉。

歲月流逝，佩吉的系統運作得完美無瑕。直到有一天，一位魔法師前來挑戰，提出了一個問題：

「你說你能容納十萬卷軸，但你的架子只有一百格。如果我同時要求調閱兩百卷軸，且這些卷軸的羊皮紙張張都是不同莊園的、從未被召回過的，你的系統會怎麼樣？」

佩吉微微一笑，答道：「我會將你的請求排隊，一次只召回一百張。當你需要用某一張時，它必已在架上；當你需要某張不在架上的羊皮紙時，我會即時召回，而你只需要稍候片刻。」

魔法師點頭讚嘆。這時，人群中一位老者低聲說道：「這不就是作業系統中那套將記憶體虛擬化、並以分頁方式映射的故事嗎？」

佩吉笑而不語，轉身繼續整理他的目錄簿。

## 概念解釋

### 虛擬記憶體佈局（Virtual Memory Layout）

Linux 系統中，每個程序（process）都擁有一個獨立的虛擬位址空間，範圍通常為 0x00000000 到 0xFFFFFFFF（32 位元），或更大的 0x0000000000000000 到 0xFFFFFFFFFFFFFFFF（64 位元）。這個虛擬空間，並不是實際的實體記憶體，而是一層抽象。

典型的程序虛擬記憶體佈局由高址到低址依序為：

```
高地圖（High Address）
+----------------------+
|     Kernel Space     |  // 核心空間，程式無法直接訪問
+----------------------+
|     Stack           |  // 堆疊區，存放函式呼叫、區域變數
+----------------------+
|     Memory Mapping   |  // 記憶體映射區（mmap、共享程式庫）
+----------------------+
|     Heap            |  // 堆積區，動態配置記憶體（malloc/new）
+----------------------+
|     BSS             |  // 未初始化全域/靜態變數
+----------------------+
|     Data            |  // 已初始化全域/靜態變數
+----------------------+
|     Text            |  // 程式碼區（唯讀）
+----------------------+
|     Reserved        |  // 保留區
+----------------------+
低地圖（Low Address）
```

### 分頁機制（Paging）

虛擬記憶體透過「分頁」機制映射到實體記憶體（RAM）或磁碟交換區（swap）。

關鍵概念：
- **分頁（Paging）**：虛擬位址空間被劃分為固定大小的區塊，稱為「分頁」（page），常見大小為 4KB。實體記憶體對應的單位則稱為「分頁框」（page frame）。
- **頁面表（Page Table）**：作業系統為每個程序維護一張頁面表，記錄每個虛擬分頁目前映射到哪個實體分頁框，或者標記為「不在記憶體中」（not present）。
- **分頁置換（Page Replacement）**：當實體記憶體不足時，作業系統會將某些較久未使用的分頁內容寫回磁碟 swap 區（俗稱「換出」），並釋放該分頁框供其他分頁使用。這就是故事中佩吉將羊皮紙歸還莊園的情節。
- **分頁缺失錯誤（Page Fault）**：當程序訪問一個尚不在實體記憶體中的分頁時，CPU 會觸發 page fault，作業系統介入將該分頁從磁碟載入實體記憶體，再恢復程序執行。

Linux 中，可用 `cat /proc/self/maps` 查看目前程序的虛擬記憶體佈局；可用 `getconf PAGE_SIZE` 查看系統分頁大小（通常為 4096 位元組）。

# 寓言故事二：雙子城的鏡影工匠

## 故事

在遙遠的北方，有一座以鏡像複製聞名的工藝城市——雙子城。這座城市的工匠有一項舉世無雙的絕技：他們能將一張藍圖，精確無誤地複製出無數份副本，且複製速度快得驚人，幾乎瞬間完成。

城市的中心有一位總工匠長，名叫福克。他是整個城市最有聲望的人，因為他掌握著一把傳說中的「複製之筆」。

福克的工作方式是這樣的：當一位委託人遞交一份原始藍圖（稱為「父卷」）並要求複製時，福克有兩種做法。

**第一種做法（笨拙的複製）**：福克立刻叫來十位工匠，將父卷的每一張紙、每一行文字、每一幅插圖統統抄寫一遍，耗時又耗力。如果委託人只是想要一份以備不時之需（最後可能根本不需要修改），這種做法簡直是極大的浪費。

**第二種做法（聰明的鏡影術）**：福克並不急於抄寫，而是取出一面特殊的水銀鏡，將父卷輕輕映在鏡面上。他宣布：「這份藍圖目前有兩個鏡像——原卷和副本——但兩者共享同一批原始紙張。我只在一方試圖在紙張上書寫時，才會真正將該紙張分離出來，給予它嶄新的紙張；另一方則繼續使用原紙張。」

換句話說：福克在複製的那一刻，只是建立了「鏡像關係」，而不是真正的紙張複製。直到其中一方（父卷或副本）需要實際修改藍圖上的內容時，才會真正複製出新的紙張。

這種技法被稱為「鏡影共享」。

---

歲月如梭，福克接受了越來越多的複製委託。然而麻煩來了。

某日，一位急躁的年輕工匠湯姆前來，他拿到了一份福克複製出來的藍圖副本後，立刻想要在上頭修改線路，重新設計整張圖紙。湯姆拿起筆就開始畫，完全沒有通知福克。這時，水銀鏡面發出了一道警示光芒——原來，湯姆正在修改的那一張紙，是父卷和其他所有副本也在使用的共享紙張！

一旦湯姆寫下去，父卷和其他副本就會同時被改變。這可不是大家想要的。

就在這千鈞一髮之際，福克衝上前去，制止了湯姆，說：「等等！在你修改之前，讓我先將這張紙分離出來，單獨給你一張嶄新的空白紙張。父卷和其他副本維持不變。」這就是「鏡影術的分離時刻」。

此時，人群中一位旁觀者突然領悟，喊道：「這不就是我們作業系統中 fork() 時採用的 Copy-on-Write 機制嗎？當父程序 fork() 出子程序時，並不會立刻完整複製整個記憶體空間，而是共享記憶體分頁；只有當其中一方實際寫入資料時，才會真正進行複製！」

福克微微一笑，說道：「正是如此。我們不是笨蛋，不會在不需要寫入時就急著複製。」

---

## 概念解釋

### fork() 系統呼叫

`fork()` 是 Unix/Linux 系統中用於建立新程序的系統呼叫。呼叫成功後，會建立一個新的子程序（child process），除了少數差異（如程序 ID、回傳值）外，子程序幾乎完全拷貝了父程序的完整位址空間，包括：

- 程式碼段（Text Segment）
- 已初始化/未初始化的資料段（BSS、Data Segment）
- 堆疊段（Stack Segment）
- 堆積段（Heap Segment）
- 開啟的檔案描述符（File Descriptors）
- 訊號處理的設定

在傳統的 Unix 實作中，`fork()` 會立刻複製父程序的整個位址空間。這在記憶體較大的程式（數百 MB 甚至數 GB）上，成本極為昂貴——不僅要複製所有資料，還要確保子程序有自己獨立的記憶體副本。

### 寫入時複製（Copy-on-Write, COW）

Copy-on-Write 是一種最佳化策略，用於大幅降低 `fork()` 的複製成本。其核心思想如下：

1. **fork() 時不複製資料**：當 `fork()` 被呼叫時，作業系統將父程序和子程序的虛擬頁面同時映射到同一組實體分頁框（page frame）。雙方的頁面表項（page table entry）被標記為「唯讀」。
2. **讀取時無需複製**：若父子程序都只是讀取資料，雙方共享同一組實體分頁，無需任何複製，效能極高。
3. **寫入時觸發真正的複製**：當其中一個程序（父或子）試圖寫入某個共享分頁時，CPU 會產生一個保護錯誤（protection fault）。作業系統介入，分配一組新的實體分頁，將原有內容拷貝過去，再將寫入方的頁面表指向新的分頁，並將其改為「可讀寫」。另一方維持對原分頁的唯讀共享。
4. **虛擬記憶體隔離**：最終，父子程序各自擁有自己的獨立分頁，寫入互不干擾，虛擬記憶體空間完全隔離。

### COW 的效益

COW 的關鍵優點在於：许多程序在 `fork()` 之後會立刻呼叫 `exec()` 載入新程式，替換掉整個位址空間——在這種情況下，fork 時的資料複製完全浪費。COW 讓「純 fork + exec」的情境几乎零成本複製，大幅提升系統效能。Linux、BSD、macOS 等現代作業系統都廣泛採用 COW 機制。

# 寓言故事三：驛站中的信使與計時器

## 故事

在大漠邊疆，有一間往來旅人必经的驛站。驛站裡有兩種最重要的服務：信使投遞，以及計時通知。

驛站的規矩是这样的：

**信使制度**：每當外頭有人要送訊息給驛站裡的某人時，信使會先將訊息寫在一張小紙條上，然後在驛站門口搖鈴。驛站的僕役聽到鈴聲，便將紙條收進一個木盒裡，等待驛站主人前來領取。這就是「訊號佇列」。

**計時器制度**：每位在驛站裡等候的旅人，可以向管理員領取一個沙漏計時器。當沙子漏完時，管理員會親自走過來提醒那位旅人。這個沙漏，就是作業系統中的「計時器訊號」。

驛站主人的名字叫阿爾德，他是一位盡責的管理者。

---

有一天，阿爾德在整理木盒時，突然想起一件事：「我上次離開時，是不是有張重要的紙條在盒子裡？我得趕快處理，否則可能會錯過什麼。」

就在阿爾德伸手去拿木盒中的紙條時，驛站外頭又響起了鈴聲——另一名信使趕到，放下了新的紙條。阿爾德正要去取新紙條時，沙漏也漏完了，管理員前來通知阿爾德。這下阿爾德同時面臨三件事：新紙條、沙漏通知、以及他正在整理的木盒。

阿爾德慌了手腳。他一邊伸手去拿新紙條，一邊還要回應管理員的沙漏提醒，同時還在想：「木盒裡現在到底有多少紙條？有幾張是我剛才看過的，有幾張是新的？我有沒有誤把某張紙條當成已處理過但實際上還沒處理？」

**問題的根源**：阿爾德試圖同時處理多件事，卻沒有一個清晰的先後順序。當他正處在「檢查木盒」這個步驟的中間狀態時，新的訊息（紙條或沙漏）可能隨時湧入，打斷他的思緒。如果他沒有做好記錄，就會遺漏、遺忘，甚至重複處理某些訊息。

這就是「競態條件」——結果取決於事件發生的順序，而這個順序在沒有同步控制的情況下，是無法預測的。

---

後來，驛站來了一位老軍師，向阿爾德傳授了一套辦法：

**第一，同步機制**：老軍師教阿爾德，每次進入「整理木盒」的流程前，先在門口放一塊「整理中」的牌子。在牌子放下期間，任何新來的紙條和沙漏通知都只能在門外等候，不得打擾。只有當阿爾德完成處理、清除牌子後，下一個等候中的事件才能進入。這就是「阻塞訊號」（signals are masked）與「臨界區」（critical section）的概念。

**第二，原子操作**：老軍師說，千萬不要在檢查到一半時停下來。如果要遍歷整個木盒，就一口氣全部看完，不要中間摻雜任何其他事件。

阿爾德照做之後，驛站再也沒有發生過混亂。他終於明白：訊號處理本身並不危險，危險的是在處理的過程中沒有阻擋新訊號的進入，導致狀態不一致。

這時，人群中一位旁觀者說道：「這不就是作業系統中，處理非同步訊號時所面臨的競態條件嗎？訊號會在任意時間點打斷程序的執行，如果處理常式（signal handler）和主程序之間共享了某些狀態，而缺乏同步機制，就會造成資料不一致。」

阿爾德點頭稱是。

---

## 概念解釋

### 訊號（Signals）在 Linux 中的特性

訊號是 Linux 中程序間非同步通訊的基本機制。每個訊號代表一種事件，諸如：
- `SIGSEGV`：無效記憶體存取
- `SIGCHLD`：子程序結束
- `SIGALRM`：計時器逾時
- `SIGINT`：中斷鍵（Ctrl+C）
- `SIGUSR1/USR2`：使用者自訂訊號

訊號的送達時間是不可預測的——取決於系統排程與事件觸發時機，程序隨時可能在任意指令邊界被中斷，進入訊號處理常式（signal handler）執行。

### 競態條件的由來

訊號處理常式（handler）與主程序之間存在一個根本性的問題：**主程序不知道訊號何時會送達**。如果 handler 與主程序同時訪問、修改某個共享變數（critical data），就會產生競態條件（Race Condition）：

```c
// 主程序
counter = counter + 1;  // 若訊號在此時送達……
// 訊號處理常式
counter = counter + 1;  // ……就會造成計數錯誤
```

在缺乏保護的情況下，兩個執行緒（或主程序與訊號 handler）對同一變數的讀-改-寫操作沒有原子性保證，最終結果可能少計或重複計數。

### 訊號處理的競態條件常見陷阱

1. **非同步訊號打斷系統函式**：如果在訊號處理常式中呼叫了非 async-signal-safe 的函式（如 `printf()`、`malloc()`、`free()`），可能導致死鎖或資料結構破壞。Linux 規定訊號 handler 中只能使用「異步訊號安全」（async-signal-safe）的函式（如 `write()`、`_exit()`）。
2. **共享狀態不一致**：訊號 handler 與主程序共享的全域變數，可能在 handler 修改後、主程序讀取之前又被其他事件修改。
3. **中斷嵌套**：訊號處理期間，如果同一訊號再次送達（預設行為下通常會阻擋，但取決於設定），可能造成嵌套執行造成狀態混亂。
4. **不完全的系統呼叫 restart**：某些系統呼叫在等待 I/O 時若被訊號中斷，作業系統預設會重啟該呼叫，但若未正確處理，返回值（-1 + errno=EINTR）需要主動判斷並重試。

### 如何避免競態條件

1. **使用 `sigprocmask()` 阻擋訊號**：在進入關鍵區段（critical section）前 block 訊號，離開後再 unblock，使處理具有原子性。
2. **使用 `sigaction()` 的 `SA_RESTART`**：控制被中斷的系統呼叫是否自動重啟。
3. **訊號處理的設計原則**：保持 handler 極度簡單——只設置一個 volatite 標記（flag），所有實際邏輯由主程序在安全時機執行（"check flag in main loop" 模式）。
4. **使用 IPC 替代複雜的訊號邏輯**：對於複雜的並發需求，優先考慮管道（pipe）、訊息佇列（message queue）或 POSIX 訊號量等更安全的機制。

### 範例：經典的非安全模式 vs 安全模式

```c
// 非安全：handler 中做太多事，且呼叫不安全函式
void bad_handler(int sig) {
    char buf[100];
    sprintf(buf, "Signal %d received\n", sig);
    printf(buf);  // 不安全：printf 不是 async-signal-safe
    global_flag = 1;
}

// 安全：handler 只設置 flag，主程序輪詢
volatile sig_atomic_t sig_flag = 0;
void safe_handler(int sig) {
    sig_flag = 1;  // 只需要一條指令，原子操作
}

int main() {
    signal(SIGINT, safe_handler);
    while (1) {
        if (sig_flag) {
            sig_flag = 0;
            handle_signal();  // 在安全時機處理
        }
        do_other_work();
    }
}
```

`volatile sig_atomic_t` 確保對該 flag 的讀寫為不可中斷的原子操作，是處理訊號標記的標準做法。

# 故事：村莊裡的守井人

從前有一座繁榮的村莊，村莊廣場中央有一口古老的水井，井水甘甜，四季不竭。

村莊裡住著十幾戶人家，每戶每天都得到井邊打水。最早的時候，村民們的做法是這樣的：

每天早晨，張大嬸提著水桶來到井邊，站在那裡等水桶沉下去、打上來、把水倒進自己的桶裡，才算完成一桶。這期間她哪裡也不能去，只能乾等。等水打滿了，她提起桶回家，才發現鄰居李大哥、王大爺、陳小哥也都各自站在井邊——原來他們也是用同樣的方法，一戶一戶輪流等，輪到誰誰才能打水。整個廣場擁滿了人，大家都站在那裡發呆，卻又不敢離開，深怕一離開自己的位置就沒了。

這種做法有個名字，叫「輪流探問法」。每戶人家派一個僕人守在井邊，不斷地問：「我的水好了嗎？」井每次只回答一個人，其他人都只能繼續等。等到天荒地老，水井確實能供應所有人，但整個村子浪費了大量的時間——僕人們站了一整天，卻大多數時候都在發呆，根本不知道自己什麼時候才能完成。

時間一久，村民們受不了了，決定改革。

他們選出一位德高望重的守井人——老陳師傅。老陳師傅不是普通村民，他的工作只有一個：同時盯著廣場上所有要打水的人。

張大嬸來了，把自己的水桶交給老陳，交代一句：「水滿了叫我。」然後她就回家了，該做飯做飯、該織布織布。

李大哥來了，同樣把水桶交給老陳：「水滿了叫我。」然後也回家了。

不一會兒，老陳師傅喊了：「張大嬸！水滿了！」張大嬸回來，提起水桶，優雅地離開。

又過了一會兒：「李大哥，好了！」李大哥回來，提起水桶，回家。

其他人根本不需要知道別人的進度，只需要在自己的水桶被喚醒時回來就好。老陳師傅一個一個喊，井邊再也不需要排隊，廣場空蕩蕩的，大家都做著自己的事，只有被叫到的人，才會出現。

後來，井打得越來越快，村民越來越多，老陳師傅開始應付不過來。於是他改良了方法：他不再一個一個盯，而是用一根長繩，同時牽著廣場上所有水桶的繩頭。哪一個繩頭抖動了，就意味著那一桶水滿了，他立刻喊對應的人來取。這就是「poll」——用一根繩子同時感測所有水桶的狀態。

再後來，村民數量暴增，繩子也開始打結了。老陳師傅乾脆把繩子換成一種特製的「震動板」——只要有任何一個水桶震動，整塊板子就會發出輕微的顫抖，聲音雖然小，但老陳師傅只要把手放在板子上，立刻就能精確地知道是哪一個水桶在震動。這就是「epoll」——最高效的監控機制，瞬間知道哪一個資料來源有回應，而且根本不需要一遍一遍去檢查每一個人。

村莊廣場從此再也不會擁擠，大家都安靜地生活著，只有被老陳師傅叫到的人，才會來到井邊。

# 概念解釋：I/O Multiplexing（多路復用 I/O 模型）

## 問題的由來

在作業系統中，當一個程序需要同時處理多個 I/O 請求時，傳統的做法是對每個 I/O 來源逐一進行「輪詢」（Polling）：向第一個檔案描述符問「你準備好了嗎？」，再問第二個、第三個……在等待期間，程序無法做其他事情，效率極低。

## select、poll、epoll 的核心思想

多路復用 I/O 的核心思想只有一個：**不要再讓程序自己一個一個去問，而是把監控的職責交給作業系統，讓作業系統告訴程序「哪一個已經準備好了」**。

### select
系統呼叫 `select()` 允許一個程序同時監控多個檔案描述符（file descriptor）。程序把感興趣的描述符集合交給核心（kernel），然後陷入睡眠。當集合中任何一個描述符就緒時，核心喚醒程序，程序再用 `FD_ISSET` 等巨集一個一個查詢「到底是哪一個」準備好了。

**限制**：監控的描述符數量有上限（`FD_SETSIZE`，通常為 1024）；每次呼叫都需要把描述符集合從使用者空間複製到核心空間，效能損耗大；而且喚醒後還是要線性掃描整個集合才能確認是哪一個。

### poll
`poll()` 的運作原理與 `select()` 類似，但採用陣列結構而非固定大小的集合，因此在描述符數量較多時不會有上限問題。但本質上仍需把整個陣列複製進核心，且喚醒後仍需線性掃描。

### epoll
`epoll()` 是 Linux 特有的高效能方案，克服了前兩者的所有缺點：

- **不複製描述符集合**：程序只需向核心註冊一次感興趣的描述符，之後核心維護該集合，無需重複複製。
- **不回傳時不複製**：平常時程序呼叫 `epoll_wait()` 進入睡眠，只有當真正有描述符就緒時才被喚醒，並且只回傳已就緒的那幾個描述符，而非整個集合。
- **時間複雜度**：核心內部使用紅黑樹管理描述符，加入、刪除、查詢均為 O(log N)，喚醒時只回傳已就緒的描述符（通常很少）。

`epoll` 的三個核心 API：
- `epoll_create()` / `epoll_create1()`：建立一個 epoll 實例
- `epoll_ctl()`：向 epoll 實例新增、修改、刪除要監控的描述符
- `epoll_wait()`：等待已監控描述符中任何一個就緒

## 實際應用場景

所有高效能的網路伺服器幾乎都基於 epoll 構建，例如 Nginx、Redis、Node.js（底層）在 Linux 上均使用 epoll。伺服器需要同時維護數萬甚至數十萬個並發連線，輪詢任何一個都是不可能的，唯有 epoll 才能高效地做到「同時監控大量資料來源，只在真正有事的時候才通知程序」。

# 故事：花園裡的令牌

山腳下有一座美麗的社區花園，住了好幾戶人家。花園裡有一座公共工具房，鋤頭、鏟子、澆水管、剪枝剪統統放在那裡，所有住戶都可以使用。花園長年生機盎然，住戶們相處融洽，直到一個春天的早晨。

那天，劉大哥清晨五點就起身了，急匆匆地奔向工具房，取了唯一的一把鋤頭，開始翻土。翻呀翻，翻到太陽高升，王大姐走進工具房，一看——鋤頭不在了。

「鋤頭呢？」王大姐急得跺腳。她的菜園子正等著翻土，今天不翻土，豆苗就趕不上播種的節氣了。

「被劉大哥借走了。」鄰居趙奶奶無奈地說。

王大姐只好等。等到中午，等到下午，等到天快黑了，劉大哥才拎著滿身泥土走回工具房，放下鋤頭。王大姐衝過去一把抓住：「終於還了！」她抓起鋤頭就往外跑，心想今天總算還來得及。

可是她不知道的是，這時李阿姨正在花園另一頭等澆水管，等了半天找不到，原來劉大哥把澆水管也借走了還沒還。整個春天，花園裡天天都有人因為工具被佔用而苦等，有人因為等太久乾脆放棄，有人因為找不到工具急得跳腳。

終於，大家再也受不了了，召集了一次花園居民大會。

## 第一個解決方案：排隊法

有人提議：「每次只能一個人借一樣工具，借完了才能輪下一個人。」大家一試，發現根本不行——劉大哥借了鋤頭要乾三個小時，期間所有人都得在工具房外面排隊等著，花園裡空無一人，時間全浪費在排隊上了。

## 第二個解決方案：令牌法

最後是老園丁阿福伯站出來，提出了他的法子。

他在工具房門口放了一個木盒子，盒子裡放著三枚木令牌。第一個格子上寫著「鋤頭」，第二個寫著「鏟子」，第三個寫著「水管」。

誰想借工具，先到盒子前看看有沒有對應的工具令牌：
- 如果有，就拿走令牌，進工具房取工具，用完後把令牌放回盒子。
- 如果沒有，就坐在花園涼亭裡等，絕不去工具房擅自拿工具。

這樣一來，大家從不抢工具了——沒有令牌就一定不能拿。有人借走了鏟子，盒子裡就不會有鏟子的令牌，其他人一看到盒子就知道「鏟子被借走了，等一下再來。」

劉大哥今天早上來了，拿走了鋤頭令牌，取了鋤頭在菜園忙活。這段時間裡，王大姐走進工具房，看到盒子——鋤頭令牌不在，她就知道有人借走了，乖乖去涼亭坐著，安心等待。

不僅如此，阿福伯還給這個盒子訂了一條規矩，寫在盒子蓋上：「任何人拿走了令牌，就等於對所有其他住戶許下了諾言：你一定會歸還。每一枚令牌，代表的是一個人對整個花園社群的承諾。」

這就是「semaphore 號誌」——一種用來表示「某項資源還有幾份」的管理機制。拿走令牌代表「我正在使用」，歸還令牌代表「我使用完了」。如果令牌拿光了，後來的人就必須等待，絕不能擅自動工具。

## 意外的考驗

可是，有一天，年輕人小楊搬進了花園。他急躁、做事不排隊，看到工具房裡有鏟子（別人忘了把鏟子還給令牌盒），就擅自拿了去用，根本沒有理會令牌盒的存在。

阿福伯見狀，嘆了一口氣說：「小楊啊，這個花園靠的不是鎖，是大家對規矩的自覺遵守。你不拿令牌就拿工具，破壞的不是鎖，是整個社群的信任。」

小楊臉紅了，此後他再也沒有違反過規定，花園重新回到了和諧的狀態。

而那座公共工具房——那就是「共用記憶體」（Shared Memory）。所有居民共享同一個空間的資源，這空間是大家共同可見的。令牌盒，則是「號誌」（Semaphore），它不鎖門，只是告訴每一個人「目前有多少資源可供使用」，讓大家自己決定是否等待。

這就是 Unix 世界裡，處理共享資源同步的經典方式：**共享記憶體提供「共用的空間」，號誌提供「共用的規則」**，兩者配合，才讓多人協作得以安全進行。

# 概念解釋：Shared Memory（共用記憶體）與 Semaphores（號誌）

## 為什麼需要共享記憶體？

在多程序（multi-process）環境中，每個程序有自己獨立的虛擬位址空間，原則上彼此無法直接讀寫對方的記憶體。但在某些場景下，多個程序需要高效地交換大量資料（例如：多個程序同時處理同一筆快取資料庫），這時效能瓶頸在於資料複製——如果每次都要透過 IPC 管道或訊息佇列來回複製，開銷極大。

共享記憶體提供了最直接、最高效的解決方案：**作業系統讓多個程序映射到同一塊實體記憶體區域**，任何一個程序寫入的資料，其他程序立刻可見，無需任何複製。Unix 系統呼叫 `shmget()` 建立共享記憶體區段，`shmat()` 將其附加到程序的位址空間，`shmdt()` 將其剝離。

## 共享記憶體的問題

共享記憶體帶來了資料競爭（race condition）的風險：當多個程序同時讀寫同一塊記憶體時，誰先寫、誰後寫、資料完整性如何維護，都沒有任何保障。這時就需要同步機制。

## 號誌（Semaphores）

號誌是一種計數器，用於表示「某項資源的可用數量」。號誌的操作只有兩種：

- **P 操作（wait / down）**：嘗試取得資源。如果計數 > 0，減去 1 並繼續執行；如果計數 == 0，程序進入睡眠，等待其他程序執行 V 操作。
- **V 操作（signal / up）**：釋放資源。計數加 1，並喚醒任何正在等待的 程序。

號誌的關鍵特性是「原子性」——P 和 V 操作在硬體層級保證不可分割，絕對不會有兩個程序同時執行 P 操作而導致計數錯誤的情況。

在 Linux 中，號誌透過 System V semaphore API（`semget()`、`semop()`、`semctl()`）或 POSIX semaphore API（`sem_init()`、`sem_wait()`、`sem_post()`）操作。

## 共享記憶體 + 號誌的經典模式

```
程序 A                          程序 B
  │                                │
  │─── sem_wait(&sem) ────────────> │  （取得號誌，進入臨界區）
  │                                │
  │─── 寫入共享記憶體 ────────────> │  （安全地讀寫共享資料）
  │                                │
  │─── sem_post(&sem) ────────────> │  （釋放號誌，退出臨界區）
```

`shmget()` + `shmat()` 建立共享記憶體區段，`semget()` + `semop()` 建立號誌集合。號誌保護的不是共享記憶體本身（它只保護一段「程式邏輯」——即臨界區 Critical Section），而是確保同一時間只有一個程序能進入臨界區，從而達成對共享資料的安全存取。

## Advisory（勸告式）而非強制

這裡有一個至關重要的細節：號誌是一種「 advisory locking」機制——作業系統不會強迫程序遵守。如果程序故意繞過號誌直接訪問共享記憶體，作業系統不會阻止，只是其他遵守規則的程序將受到干擾。因此，**所有參與者都必須自願遵守號誌的規則**，這與下一個概念即將提到的「檔案鎖定」有直接關聯。

# 故事：圖書館的閱讀室

在一座古老的城鎮裡，有一座美麗的公共圖書館。圖書館的閱讀室裡陳列著各種珍貴的手抄書，其中一本，名為《城鎮大事記》的手抄本最為珍貴——它是全鎮歷史的記載，由鎮上唯一的書法家費時三年抄寫而成，每一頁都是獨一無二的藝術品。

閱讀室的規矩很簡單，寫在入口處的一塊木牌上：

> 「請有興趣閱讀《大事記》的讀者，在閱讀室的門口登記簿上留下姓名與借閱時段。登記完畢後，方可取書閱讀。閱讀完畢，請將書放回原處，並在登記簿上劃去自己的名字。」

這是一種「榮譽制」的借閱方式，靠的是每位讀者的自覺。

## 第一個春天：和諧運轉

那年春天，鎮上的讀書人還算守規矩。張書生來了，在登記簿上找到一格空位，寫下「張書生，卯時至午時」，然後取書，安靜閱讀。午時一到，他把書放回原處，在自己的名字上畫了一條線。

李秀才來了，看到《大事記》還在架上，再看登記簿，張書生的名字已經劃掉了，他便拿起筆，寫下「李秀才，午時至申時」。接著取書，細細品讀。

一切運作良好。大家都看一眼登記簿，就知道書有沒有人借；借到了就安心閱讀，沒借到就先去圖書館其他角落翻翻別的書。

這種做法有個名字——「諮詢式鎖定」（Advisory Locking）。登記簿本身沒有任何力量來阻止不守規矩的人取書，它只是提供資訊：「這本書目前有沒有人借」。讀者之間自願遵守這項資訊的指引，和諧地共享資源。

## 第二個春天：裂痕初現

可是，有一天，來了一位性子急躁的年輕人，阿德。

阿德走進閱讀室，看到《大事記》擺在架上。他沒有理會入口處的登記簿，直接伸手就取了書，坐下來開始閱讀。

這時，張書生正好也在圖書館裡。他走到門口，看了一眼登記簿——空著的，沒有人借。阿德明明正在讀書，可是他沒有在登記簿上寫字，張書生根本不知道書被借走了。

張書生以為書沒人借，便也取了一本《大事記》——圖書館恰好有另一個複製品。兩人各讀各的，相安无事。直到費書法家走進來，臉色鐵青：「你們兩個人同時閱讀，萬一把書頁撕壞了怎麼辦？這可是唯一的一本手抄本！」

阿德不以為然地說：「那個登記簿又不是什麼結實的門鎖，它只是塊木頭牌子，寫了字又有什麼用？該讀就讀，難道木牌還能阻止我？」

張書生啞口無言。因為阿德說的是事實——登記簿沒有任何強制力。

## 第三個春天：一把真正的鎖

費書法家下定決心，要徹底解決這個問題。

他在《大事記》的書架上裝了一把精巧的黃銅鎖。這把鎖與眾不同：它有兩把鑰匙，一把是「閱讀鑰匙」，一把是「管理鑰匙」。管理鑰匙由費書法家自己保管，任何人要閱讀，必須向他借「閱讀鑰匙」，拿了鑰匙才能開鎖取書，閱讀完畢還了鑰匙，鎖就自動落下。

不僅如此，這把鎖還有一個特性：如果有人試圖在書被鎖住的時候強行開鎖，鎖會發出尖銳的聲響，同時卡住整個書架，直到管理員趕來處理為止。任何人，不管知不知道鎖的存在，只要書被鎖住了，就根本不可能不透過鑰匙取出書來。

現在，不管性子急躁的阿德怎麼折騰，只要書被別人借走了、鎖落下了，他就算把整個閱讀室翻了個底朝天，也不可能拿到書。要麼等待，要麼去找費書法家協商——但他沒有任何辦法繞過鎖本身。

這就是「強制鎖定」（Mandatory Locking）。作業系統在核心層級實作了鎖，無論程序是否知道鎖的存在、是否願意遵守，作業系統都會強制阻止同時存取。

可是，費書法家也發現了新問題：要安裝這把鎖，代價很大。每次借還書都需要管理員親自操作，而且如果書被借走的時間很長，後面等書的人只能乾等，甚至管理員也沒有辦法臨時中斷借閱。更重要的是，並非所有資源都能用這種方式加鎖——鎖本身也需要消耗系統資源。

## 最終的選擇

經過慎重考慮，費書法家決定兩種方式並用：對於《大事記》這樣的無可取代的珍本，使用強制鎖定，確保萬無一失；而對於圖書館裡其他普通的書籍，仍採用原來的登記簿方式，因為那些書有很多複製品，而且損壞了可以修復，沒有必要用這麼重的代價來保護。

圖書館終於找到了適合的平衡，而那個登記簿——也就是諮詢式鎖定——依然掛在閱讀室門口，只是現在每個人都更深刻地理解到：**它的價值不在於它能阻止什麼，而在於每一個參與者都選擇去遵守它**。

# 概念解釋：File Locking（檔案鎖定）與 Advisory Locking（諮詢式鎖定）

## 檔案鎖定是什麼？

檔案鎖定是一種協調多個程序對同一個檔案進行存取的機制。當多個程序可能同時讀寫同一個檔案時，如果沒有任何協調機制，就會發生資料損壞（例如：兩個程序同時 append，一方的資料可能被另一方覆蓋）。檔案鎖定讓程序可以在開始操作前「宣告」自己對檔案某個區域的獨占權，作業系統或程式框架則確保這種獨占權得到維護。

## Advisory Locking（諮詢式鎖定）

在 Unix/Linux 傳統中，檔案鎖定幾乎都是「 advisory locking」。作業系統提供 `flock()` 系統呼叫和 `fcntl()` 的鎖定功能，程序可以透過這些 API 對檔案加鎖（取得鎖）或解鎖（釋放鎖）。作業系統會記錄誰持有哪些鎖，並在有衝突的加鎖請求出現時讓後來的程序等待或直接回傳錯誤。

**但關鍵在於**：作業系統並不會阻止一個程序繞過鎖定機制、直接用 `open()` + `read()` / `write()` 訪問檔案。如果程序 B 不調用 `flock()` 或 `fcntl()` 來請求鎖，它完全可以無視程序 A 正在持有的鎖，直接讀寫檔案——作業系統不會阻止。

這就像故事中的登記簿：它是一個「公告板」，告訴遵規守紀的人「這本書有人借了」。但它不會封住書架的大門，不守規矩的人完全可以繞過它。**所有參與者都必須自願合作，鎖定機制才能發揮作用**。

Linux 的 `flock()` 提供一種簡單的「勸告式」檔案鎖，可套用於整個檔案；`fcntl()` 则可以在記錄層級（record-level）加鎖，精確到檔案的某一個位元組區間（`F_GETLK`、`F_SETLK`、`F_SETLKW`）。

## Mandatory Locking（強制鎖定）

與 advisory locking 對應的是 mandatory locking，作業系統在核心層級強制執行鎖定，即使程序不知道有鎖，也無法突破它。在 Linux 中，要啟用 mandatory locking，必須滿足以下條件：

1. 檔案所在檔案系統必須以 `mand` 選項掛載（多數現代 Linux 發行版的預設值是 `nomand`，即不支援）
2. 檔案本身必須設定 set-group-ID 位元且 группа執行位元被清除（`chmod g+s,g-x <file>`），作業系統才會在該檔案上強制執行鎖

在實務上，**mandatory locking 在 Linux 生態系統中幾乎從未被使用**。原因是：
- 效能開銷大：每次檔案存取都需要核心檢查鎖狀態
- 可攜性差：Solaris、BSD 等其他 Unix 系統並不保證支援 mandatory locking
- 彈性不足：強制的鎖無法讓程序在緊急情況下協商例外處理

因此，在真實世界的 Unix/Linux 應用中，幾乎所有的檔案鎖定都是 advisory locking，`flock()` 和 `fcntl()` 是標準工具。應用程式必須自己約定「所有程序在寫入之前都必須先 `flock()` 或 `fcntl()` 加鎖」，才能確保安全——這是一種社會契約，而非物理柵欄。

## 與 Shared Memory / Semaphore 的比較

| 特性 | Advisory Locking | Shared Memory + Semaphore |
|---|---|---|
| 保護對象 | 檔案 | 記憶體區段 |
| 強制力 | 無（靠自願遵守） | 無（號誌也是 advisory） |
| 應用層級 | 作業系統檔案層 | 記憶體資料 |
| 常見 API | `flock()`, `fcntl()` | `shmget()`, `semop()` |
| 適用場景 | 多程序寫入同一檔案 | 多程序共享大量資料 |

## 實務建議

在 Linux 上，多程序共享資料時的標準實踐是：

1. 如果共享的是「檔案內容」：使用 advisory file locking（`flock()` 或 `fcntl()`），並在應用層約定所有程序都必須遵守加鎖約定。
2. 如果共享的是「大量資料」：使用共享記憶體 + POSIX 信號量（semaphore），同樣依賴所有程序的自願遵守。
3. 如果需要更嚴格的同步控制：考慮使用具名管道（FIFO）或 Unix Domain Socket 進行訊息傳遞，讓一個管理程序統一協調所有存取。

在所有這些場景中，「合作意願」都是系統能正常運作的基礎——這正是 advisory locking 機制的核心寓意。

# 寓言七：共享工具屋的奇蹟

從前，在一座繁榮的村莊裡，住著許多木匠。

每個木匠都有自己的作坊，作坊裡堆滿了各式各樣的工具——鋸子、刨子、鑿子、鐵鎚、釘子、砂紙。這些工具造價昂貴，一把好鋸子要用上好的鋼材，一個精準的刨子要請村裡最好的鐵匠打造。每一個木匠都想辦法把工具備齊，但常常力不從心——有的木匠只有鋸子沒有刨子，有的木匠工具齊全卻磨損了卻無力更換。

這座村莊叫做「工匠村」，木匠之間常互相借用工具。但問題來了：借來借去，工具最後總是不知道去了哪裡；有時木匠正在用一把刨子，另一個人突然跑來拿走，刨了一半的木材就這樣報廢了。

**共享工具屋的誕生**

村裡最老的木匠師傅——大家都叫他老周的——決心改變這個局面。

他在村子中央建了一座寬敞的石屋，取名「共享工具屋」。石屋的設計很特別：四面牆上有一排排整整齊齊的架子，每個架子都有標籤，標明這是哪一種工具。工具屋有一扇厚重的門，門上掛著一把大鎖，而老周是唯一有鑰匙的人。

規則很簡單：任何木匠要使用工具，必須先到老周那裡登記，說明要用什麼工具、要多久。老周打開工具屋，取出對應的工具，交給木匠，但工具並不是「給」他——木匠只是「借用」。用完之後，木匠必須把工具還給老周，由老周放回原位，鑰匙仍由老周保管。

一開始，木匠們抱怨連連。「為什麼不能直接把工具拿走？」有人不滿，「每次借還都要找老周，太麻煩了！」

**看不見的魔術**

然而，隨著時間推移，木匠們發現了共享工具屋不可思議的好處。

首先是成本的節省。以前每個木匠都要自己想辦法弄到所有工具，現在他們只需要借用就好。一把好鋸子的費用，十個木匠分攤，每人只需出一成。

其次是維護的問題。過去每個木匠自己保養工具，保養得好不好參差不齊。現在老周負責所有的工具維護，他會定期檢查每一把鋸子的齒是否鋒利、每一個刨子的刀片是否平整。木匠們發現工具屋裡的工具，總是比過去自己保管的還要鋒利好用。

更神奇的是，當一個木匠正在用一把刨子時，另一個木匠依然可以借到另一把刨子——只要工具屋裡還有備用的。**同一把工具，理論上可以被無數個木匠同時「共享」**——只不過每個人拿到的是真正屬於自己的那一把，而不是真的在同一把工具上工作。

**意外的發現**

直到有一天，一個年輕的木匠小楊，在使用工具屋的過程中，觀察到了一個驚人的現象。

小楊仔細研究老周的記錄本，發現老周在「借出」工具時，其實從來沒有真的把工具從架子上取下來交給木匠——他只是給了木匠一張「工具借據」。借據上寫明了工具的種類和大約位置。小楊正要質問，老周笑著說：「你拿到借據，就去架子那裡取。如果你要的刨子正巧在架子上，你就直接拿走。如果不在——」

老周指了指門後的一間小房間：「那我就得去『中央工具庫』把它調來。中央工具庫是另一批工匠管理的，和我的工具屋是兩回事。不過你不用管這些，你的借據足以讓你從任何一個地方取得工具。」

小楊愣住了：「可是，我明明只是在向你的工具屋借工具，為什麼最後可能是從別的地方來的？」

老周微笑：「這就是奧妙。**你以為你在向我借工具，但其實我只管借據；真正的工具可能來自工具屋的架子，也可能來自很遠的中央倉庫——但你根本不在乎，你只在乎拿到一把能用的刨子。**」

**真相揭曉**

小楊後來才明白，這座共享工具屋，其實有兩套系統在運作：

第一套是**老周的借據系統**。這套系統在程式裡叫做「連結器」（linker）。木匠說「我需要一把刨子」，老周就給他一張借據，上面蓋著村裡的公印。這張借據保證木匠在程式中遇到「刨子」這個名字時，可以兌換成一把真正的工具。

第二套是**真正的工具調度系統**。這套系統在程式裡叫做「載入器」（loader）。有時候工具屋的架子上正好有現成的（這叫「靜態連結」，工具早就準備好了），有時候老周必須派人去中央倉庫取（這叫「動態載入」，需要的時候才去取）。

而「共享」的真正意思是：**同一把刨子的圖紙（即工具的設計藍圖）只需要一份存在工具屋裡，但每一個借了刨子的木匠，拿到的是一個「可用的實體」，而不是共享同一個物理刨子。** 只不過，如果是「靜態」的預先複製，每個木匠都會拿到自己的一把；如果是「動態共享」，則多個木匠在某些條件下真的可以共享同一個記憶體中的工具藍圖——但這種共享必須非常小心，否則就會像過去一樣混亂。

小楊終於懂了：**共享工具屋的核心，不是把工具直接分給所有人，而是在「需要時」才把工具送到需要的人手中，並保證每個人都有一個有效的工具可用。** 老周的角色，是整個系統的協調者——他既負責「讓人知道工具在哪裡」（符號解析），也負責「把工具真的送到人手上」（重定位與載入）。

## 概念解釋：動態連結與共享函式庫的加載機制

### 什麼是動態連結？

動態連結（Dynamic Linking）是一種連結方式，在這種方式下，程式的某些依賴（如函式庫）不是在程式執行前就全部綁定好，而是推遲到**程式執行時（runtime）才進行解析與載入**。

在寓言中，每個木匠（程式）在出發工作前，不需要把所有工具（函式）都先打包帶走，而是可以在需要時向共享工具屋（共享函式庫）借用。借據對應的就是「符號解析」——解決程式中「刨子」這個名字指向哪一把真正的刨子。

### 共享函式庫（Shared Libraries / .so 檔案）

在 Linux 系統中，共享函式庫的副檔名是 `.so`（shared object）。多個執行中的程式可以**共同映射同一份實體的 `.so` 檔案到各自的虛擬位址空間**。這就像多個木匠同時借用工具屋裡的同一把刨子——他們各自拿到一個「使用權」，而物理上只有一把刨子存在。

這種共享帶來了兩個核心優點：

**記憶體節省**：如果一百個程式都依賴同一個 `libc.so`，作業系統只需要在實體記憶體中保留一份 `libc.so` 的程式碼段，各個程式在虛擬位址空間中映射到同一個實體頁面。這就是「共享唯讀頁面」（shared read-only pages）。

**維護便利**：當 `libc.so` 更新了安全性修補時，所有依賴它的程式立即受益，不需要重新編譯或重新連結。

### 動態連結器（ld-linux.so）

Linux 的動態連結器是 `ld-linux.so`（或 `ld.so`）。它的職責包括：

1. **符號解析（Symbol Resolution）**：當程式參照某個函式（如 `printf`），動態連結器在所有已載入的共享函式庫中搜尋這個符號，找到後將其位址填入程式的 GOT（Global Offset Table）。

2. **重定位（Relocation）**：共享函式庫被載入記憶體後，其中的相對位址引用需要根據實際載入位置進行調整。這一步在動態連結的場景下稱為「重定位」。

3. **延遲載入（Lazy Loading）**：預設情況下，符號解析只在符號第一次被使用時才進行，而非程式一啟動就全部解析完畢。這可以縮短程式的啟動時間。

### 程式執行時的載入流程

1. **程式啟動**：kernel 在執行一個 ELF 二進位檔案時，首先會呼叫動態連結器（由 ELF 的 `PT_INTERP` 段指定），而不是直接執行應用程式的 entry point。

2. **依賴圖遍歷**：動態連結器讀取程式的 `.dynamic` 段，獲取所有需要的共享函式庫名稱（DT_NEEDED）。

3. **搜尋路徑**：依序在 `/etc/ld.so.cache`（快取）、`LD_LIBRARY_PATH`、`/lib`、`/usr/lib` 等路徑中尋找共享函式庫。

4. **mmap 映射**：找到後，使用 `mmap()` 將共享函式庫的各段映射到記憶體，並設定權限（唯讀、可讀可寫等）。

5. **符號解析與 PLT/GOT**：對於每個外部參照，動態連結器在對應共享函式庫的符號表中查詢，並填入 PLT（Procedure Linkage Table）或 GOT（Global Offset Table）。後續呼叫時，透過 PLT 跳轉，不再經過連結器。

6. **初始化**：呼叫共享函式庫的 `_init()`（或透過 `__attribute__((constructor))` 標記的函式）執行初始化。

7. **轉交控制權**：完成所有連結後，將控制權交還給應用程式的 entry point（`_start`）。

### 延遲連結（Lazy Linking）的機制

動態連結預設採用延遲連結（lazy linking），即只有在函式第一次被呼叫時才進行符號解析。這是由 PLT 機制實現的：

- 第一次呼叫 `printf`：跳轉到 PLT 中的 entry，觸發動態連結器的符號解析，將 `printf` 的實際位址寫入 GOT。
- 之後呼叫 `printf`：直接透過 GOT 跳轉到已解析的位址，跳過動態連結器。

### LD_PRELOAD 與 LD_LIBRARY_PATH

- `LD_PRELOAD`：在所有其他共享函式庫之前載入，可用於覆寫特定函式（如除錯工具、效能分析工具）。
- `LD_LIBRARY_PATH`：在預設路徑之前搜尋共享函式庫的位置，方便開發時測試本地編譯的函式庫。

### 總結

動態連結與共享函式庫是 Linux 系統中一種精巧的協作機制：**多個程式在執行時共享同一份實體的函式庫程式碼，節省記憶體；動態連結器在需要時自動將符號解析與重定位完成，使系統得以在執行期彈性組合不同的模組。** 就像共享工具屋讓木匠們不必各自攜帶全套工具，動態連結讓程式不必將所有函式碼靜態包入自身，使得整個系統更節省、更靈活、更好維護。

# 寓言八：萬國翻譯官與通關密語

從前在一片遼闊的大陸上，存在著許多互不相同的王國。

每個王國都有自己的語言、自己的制度、自己管理領土的方式。有的王國靠河流吃飯，擅長航運；有的王國在山區，擅長採礦；有的王國以農業為主，儲藏著大量的糧食。不同王國之間想做生意、互通有無，卻遇到了極大的困難——語言不通，制度各異，連最基本的「打開城門」這件事，每個王國的做法都不一樣。

有的王國用巨石堵門，要二十個人合力才能推開；有的王國用木柵欄，喊一聲口令就能開；有的王國乾脆沒有門，只有一道繩索做標記。更糟糕的是，每個王國對「什麼是文件」「什麼是目錄」「什麼是連結」的理解都完全不同。山區王國把礦脈叫做「檔案」，把礦層結構叫做「目錄」；河邊王國把船隻航行路線叫做「檔案」，把港口叫做「目錄」。如果一個商人同時和多個王國打交道，他必須學會每個王國的語言和制度，痛苦不堪。

**翻譯官的誕生**

就在各王國快要斷絕來往之際，一位名叫賽琳娜的年輕女子出現了。

賽琳娜是個天生的語言學家，她精通各國語言，而且有一個奇特的天賦：她能把任何事物的本質，轉化成一套通用的描述方式。

賽琳娜的計畫是這樣的：她在每個王國的邊境建立一個「翻譯驛站」。這個驛站裡有一位雙語侍從，侍從會說兩種語言：一邊是「通用語」，另一邊是那個王國的本地語言。

商人想要和某個王國做生意，不需要學會那個王國的語言，他只需要用「通用語」跟翻譯驛站的侍從說話，侍從就會把他的需求轉換成王國能理解的本地語言。

不僅如此，商人也不需要了解那個王國的內部制度。**他只需要知道一件事：我要處理一個「檔案」，或者我要處理一個「目錄」。** 剩下的，翻譯驛站會幫他處理。

**通用的世界觀**

賽琳娜為所有王國制定了一套「通用世界觀」：

在這套世界觀裡，**所有的事物都是「檔案」（file）**。河流是檔案、礦脈是檔案、糧倉是檔案、航行路線也是檔案。這套世界觀把世間萬物分成了幾類：

- **普通檔案**：可以讀寫的內容，比如一本帳簿、一張地圖。
- **目錄檔案**：裝著其他檔案的容器，比如一個倉庫裡的貨架清單。
- **特殊檔案**：代表設備的檔案，比如通往港口的閘門、礦坑的入口。
- **連結檔案**：一個檔案的另一個名字，比如一個港口的別名。

每一個翻譯驛站都必須把這些概念，轉換成自己王國能理解的對應事物。港口王國的侍從把「普通檔案」翻譯成「可以裝卸貨物的船位」；山區王國的侍從把「普通檔案」翻譯成「可以開採的礦層」；農業王國的侍從把「普通檔案」翻譯成「可以存取的糧倉隔間」。

**打開一扇門的藝術**

商人張三想要從港口王國取一批木材，賣到山區王國去。

他來到港口王國的翻譯驛站，用通用語說：「我要**打開**一個檔案，叫『木材清單』。」

驛站侍從聽懂了，用港口王國的本地語言對守門人說：「請拉起三號船位的吊閘。」守門人執行了命令，木材從船上卸下來，張三拿到了木材。

然後張三前往山區王國的驛站，同樣用通用語說：「我要**創建**一個檔案，叫『新到的木材』，把剛才拿到的內容放進去。」

山區王國的侍從用山區王國的語言對礦坑管理者說：「在東區三號礦道建立一個新的礦層存放區，把這批礦石存進去。」管理者執行了命令。

在這個過程中，張三完全不知道港口王國用的是「船位」和「吊閘」，也不知道山區王國用的是「礦道」和「礦層」。他只使用了「打開檔案」和「創建檔案」這兩個通用操作，而翻譯驛站負責把這些通用操作轉換成各王國實際能執行的本地命令。

**看不見的牆**

然而，漸漸地，有人發現了一個問題。

有些商人開始質疑：「這個翻譯驛站系統，是不是真的在工作？為什麼我每次說『打開檔案』，都能得到結果，但我從來沒見過那些翻譯官？」

原來，翻譯驛站的侍從大多數時候根本不在現場。**真正的翻譯官，是一套寫好的規則書，藏在驛站的地底下，沒有人看得見。** 商人們稱之為「看不見的牆」（hidden layer）。

商人走進驛站，他的「打開檔案」指令首先遇到「看不見的牆」——也就是「虛擬層」。這層負責把通用指令翻譯成對應王國的操作。翻譯的過程由侍從（或規則書）完成，商人完全不需要知道他在和哪個王國打交道，也不需要知道翻譯過程有多複雜。

**真相大白**

後來，有一位好奇的學者——叫做艾文的——決定研究這些翻譯驛站的運作方式。

艾文發現，每一個驛站底下都藏著厚厚一本《通用操作手冊》，手冊上列出了所有可能的通用操作，以及每個王國對應的本地實作方式。比如：

- **打開（open）**：港口王國→拉起吊閘；山區王國→清除坑道障礙；農業王國→打開穀倉側門。
- **讀取（read）**：港口王國→清點船艙；山區王國→測量礦層厚度；農業王國→計算穀倉存貨。
- **寫入（write）**：港口王國→裝卸貨物；山區王國→開採新礦層；農業王國→存入新糧食。

艾文恍然大悟：**這套《通用操作手冊》，就是作業系統中最重要的抽象層之一——它讓所有的「底層王國」（真實的檔案系統）都透過同一套「通用語」（統一的系統呼叫介面）對外提供服務，而商人（應用程式）根本不需要知道底層究竟是 ext4、NTFS 還是網路檔案系統。**

這就是 VFS（Virtual File System）的核心精神：**提供一個一致的介面，讓所有不同的底層檔案系統都能以相同的方式被訪問，應用程式無需關心實際的檔案系統類型。**

## 概念解釋：Linux 虛擬檔案系統（VFS）層級架構

### VFS 是什麼？

VFS（Virtual File System）是 Linux 核心中的一個軟體層，它**存在於應用程式與實際的檔案系統實作之間**，提供一套統一的、抽象的檔案系統介面。

在寓言中，翻譯驛站對應的就是 VFS 層。「看不見的牆」就是 VFS 本身——它把應用程式的系統呼叫（`open`、`read`、`write`、`stat` 等）轉換成實際檔案系統的底層操作（ext4、xfs、btrfs、ntfs、vfat 等）。

### 核心抽象：超級區塊、inode 與目錄項

VFS 定義了幾個核心的資料結構，來統一描述所有檔案系統中的檔案與結構：

1. **超級區塊（super_block）**：代表整個已掛載的檔案系統。每個掛載的分割區都對應一個 `super_block` 結構，裡面存放檔案系統的元資料（如區塊大小、總節點數、可用空間等）。

2. **inode（index node）**：代表檔案系統中的一個「檔案」。每個檔案在磁碟上有一個對應的 inode，儲存檔案的元資料（大小、權限、時間戳、資料區塊位置等）。**一個 inode 對應一個檔案，無論這個檔案是普通檔案、目錄、還是特殊檔案（裝置、連結）。**

3. **目錄項（dentry）**：代表路徑中的一個組成部分（如 `/home/user/file.txt` 中的 `home`、`user`、`file.txt`）。dentry 構成了一個名稱到 inode 的映射快取，讓路徑解析（pathname lookup）更有效率。VFS 會在記憶體中維護一個 dentry cache，緩存最近訪問過的路徑解析結果。

4. **檔案（file）**：代表一個**已開啟的檔案實例**。一個 inode 可以對應多個 `file` 結構（多個程序或同一程序多次開啟同一檔案）。`file` 結構儲存檔案在行程中的狀態（如讀寫指標位置、檔案描述元）。

### VFS 的層級架構

```
應用程式（open, read, write, ...）
        │
        ▼
    VFS 層（統一的系統呼叫介面）
    super_block / inode / dentry / file
        │
   ┌────┴────┐
   ▼         ▼
 ext4      xfs       btrfs       ntfs       ...（實際的檔案系統驅動）
   │         │         │           │
   ▼         ▼         ▼           ▼
 磁碟分割   磁碟分割   磁碟分割   網路芳鄰（samba）
```

- **應用程式層**：所有對檔案的操作都透過一致的系統呼叫介面（POSIX 介面）進行。
- **VFS 層**：將系統呼叫路由到正確的檔案系統驅動程式，並緩存元資料。
- **檔案系統驅動層**：每種實際檔案系統（ext4、xfs、btrfs、vfat、ntfs、nfs、cifs 等）各自實作 VFS 定義的介面，操作磁碟或網路上的實際資料。

### 關鍵操作：路徑解析與快取

VFS 最核心的功能之一是**路徑解析（pathname lookup）**。當應用程式執行 `open(/home/user/file.txt, O_RDONLY)` 時：

1. VFS 解析 `/`，找到根目錄的 inode（從 `super_block` 的根指標取得）。
2. 解析 `home`，在根目錄 inode 的內容中找到 `home` 的 dentry，再找到其 inode。
3. 解析 `user`，在 `home` 目錄的 inode 中找到 `user` 的 dentry，依此類推。
4. 最後找到 `file.txt` 的 inode。
5. 分配一個 `file` 結構，建立 inode 與 file 的關聯，返回檔案描述符。

這整個解析過程的結果會被寫入 **dentry cache**，因此下次再訪問同一路徑時，速度會大幅提升。

### 統一的介面如何橋接多樣的實作

VFS 定義了一套「檔案系統操作函式指標結構」，稱為 `file_system_type` 和 `inode_operations`、`file_operations` 等。每一種實際檔案系統在註冊時，都需要提供這些函式指標的具體實作。

例如，ext4 的 `file_operations.read` 可能直接操作磁碟區塊，而 NFS 的 `file_operations.read` 可能會發送網路請求到遠端伺服器。**對 VFS 層以及上層應用程式而言，這兩種實作的行為完全相同——因為它們都遵從相同的介面合約。**

### 常見 VFS 概念對照表

| VFS 概念 | 在寓言中的對應 | 說明 |
|---------|------------|------|
| VFS 層 | 翻譯驛站 | 統一介面的轉換層 |
| 應用程式 | 商人 | 只使用通用語，不了解底層 |
| 實際檔案系統 | 各王國 | ext4、xfs、btrfs、NFS 等 |
| super_block | 王國的整體制度 | 代表整個檔案系統 |
| inode | 一個檔案（無論什麼類型） | 檔案的元資料與資料區塊位置 |
| dentry | 路徑中的每一段名稱 | 名稱到 inode 的映射快取 |
| file | 一次開啟的實例 | 程序的讀寫狀態（指標、模式）|
| 通用操作（open/read/write）| 通用語指令 | POSIX 介面 |

### 為什麼 VFS 如此重要？

沒有 VFS，應用程式就必須為每種檔案系統撰寫不同的程式碼。想在不同檔案系統之間遷移資料，或者同時讀取網路磁碟和本地磁碟，開發者必須處理各種底層差異。

有了 VFS 之後，**所有檔案系統在應用程式眼中都是同一種東西**：一個可以打開、讀取、寫入、查詢屬性的統一實體。這使得諸如 `cp`、`mv`、`tar`、`grep` 這類工具可以不加修改地工作在 ext4、xfs、btrfs、NFS、tmpfs 等任何檔案系統之上。

### 總結

Linux VFS 就像寓言中的萬國翻譯官系統：**它為所有不同的底層檔案系統建立了一套通用的「世界觀」和「操作語言」，讓應用程式可以用統一的方式操作任何檔案系統中的檔案，而不需要知道底層究竟是什麼。** inode、super_block、dentry 這些資料結構，是 VFS 用來統一描述所有檔案系統的基石；路徑解析與各層快取，則是 VFS 確保這套抽象不損耗效能的關鍵機制。

# 寓言九：磨坊裡的齒輪與順序問題

從前在一個山谷裡，有一座古老的磨坊。

這座磨坊叫做「共鳴磨坊」——它不是普通的磨坊，而是一座為整個村莊服務的多功能工坊。磨坊裡有一個巨大的齒輪組，用來磨麥子、抽水、鋸木頭、甚至發電。每個村民都可以來到磨坊，搖動一個把手，讓齒輪帶動不同的工具完成工作。

磨坊的齒輪組非常精密。任何一個齒輪在轉動時，都會帶動其他齒輪。這些齒輪用皮帶和連桿相互連接，形成了一個錯綜複雜的系統。

村裡的人一開始都很滿意——磨坊效率高，大家都能用。

**問題的起源**

直到有一天，村民老陳發現了一件奇怪的事。

老陳是一個木匠，他需要用磨坊的鋸木頭工具。他搖動把手讓齒輪轉動，鋸子開始工作。但就在鋸子快要切斷木頭的時候，木頭突然被一股奇怪的力量往前推了一下，鋸子偏離了軌道，傷了老陳的手。

老陳查看現場，發現原來是另一個村民——阿花——同時在用抽水工具。她搖動了自己的把手，抽水機開始運轉，抽水管把水往外抽。但就在老陳的鋸子快完成工作時，阿花的把手突然被卡住，她使勁一拉——**這股力道透過齒輪組傳導，正好把老陳的木頭往前推了一下**。

老陳怒氣沖沖地找到磨坊管理員老周質問：「為什麼我用磨坊的時候，別人的操作會影響我？」

老周一臉無奈：「磨坊的齒輪是共用的。你轉動齒輪，影響的是整個系統，不是只有你要用的那一部分。」

**第一個嘗試：分開的齒輪**

阿花和幾個村民決定自己解決問題。他們湊錢買了一批新的皮帶和齒輪，把磨坊的一部分改造成了「獨立齒輪組」——抽水機的齒輪組和鋸木頭的齒輪組完全分開，各自運作，不再共享皮帶。

改造完成後，效果立竿見影。老陳用鋸子的時候，阿花抽水完全不受影響；阿花抽水的時候，老陳鋸木頭絲毫不受干擾。

村民們稱之為「隔離方案」。**每個工具都有自己專用的齒輪組，齒輪之間不再相互拉扯。**

但問題是，磨坊的空間有限，齒輪昂貴，不可能為每個工具都配一套獨立的齒輪組。磨麥子的齒輪、發電的齒輪、打油的齒輪……這些仍然共享同一套齒輪組。更糟糕的是，隔離方案帶來了新的問題：當老陳正在鋸木頭時，磨坊突然有人喊「緊急！需要發電！」——但發電的齒輪組此刻正被老陳占用著，別人根本無法使用。

**第二個嘗試：順序排程**

老周提出了一個新方案：**制定使用規則，統一排程**。

老周在磨坊門口設立了一本登記簿。任何村民要使用磨坊，必須先在登記簿上簽名，說明要用多久。老周負責排程，確保同一時間只有一個人使用共享的齒輪組。

村民小楊是第一批使用這個系統的人。他需要磨麥子，預計需要一個小時。他在登記簿上簽名，老周說：「好，你從上午九點開始用，下午兩點前歸還。」

在這一個小時內，其他村民不能使用共享齒輪組。他們可以排隊等候，或者去做其他事情。

這個方案解決了「衝突」的問題——同一時間只有一個人在操作齒輪組，不可能出現老陳和阿花同時操作、相互干擾的情況。

**但新的問題又來了。**

村民阿美需要磨一個小時的麥子，但她只需要五分鐘就完成了前面五分鐘的準備工作（清理麥子、調整轉速），剩下的時間她只是在等磨坊自動磨麥子。按照新規則，她必須持有磨坊的使用權整整一個小時，即使她在那段時間裡根本不需要操作任何把手。

同時，小楊正在等待磨坊空出來，但他需要做的事情只需要五分鐘。他願意等五分鐘，但阿美卻占用了整個小時。

老周意識到：**問題的根源是「使用權」的顆粒度太大**——他把整個磨坊的使用權分配給一個人，但這個人並不是全程都在操作，他只是在某些時刻操作，某些時刻在等待磨坊自動運轉。

**第三個嘗試：操作鎖**

就在這時，一位從外地來的工程師路過，給了村民們一個新點子。

工程師說：「問題不是『誰持有整個磨坊』，而是『誰在操作把手』。**你需要的不是一個人占有一整個磨坊，而是確保同一時間只有一個人在操作齒輪組的關鍵節點。**」

他設計了一種特殊的鎖——**操作鎖**，它不是鎖在整個磨坊的大門上，而是鎖在齒輪組的關鍵節點上。**只有拿到操作鎖的人，才可以操作齒輪組的核心部分。如果有人已經拿了操作鎖，另一个人必須等待。**

不僅如此，工程師還做了一件更聰明的事：**他把磨坊的操作分成了「安全區」和「危險區」**。

「安全區」是那些可以被多個人同時使用的部分。比如讀取磨坊的狀態（今天磨坊是否運轉正常），或者查看排程表（查詢下一個可用時段）。這些操作不需要操作鎖。

「危險區」是那些修改系統狀態的操作。比如啟動齒輪、改變轉速、調整皮帶張力。只有拿到操作鎖的人才能進入危險區，而且**一旦完成修改，必須立刻釋放操作鎖**。

這個設計的關鍵在於：**操作鎖保護的是「操作行為」本身，而不是「整個磨坊」。** 任何人都可以隨時進入安全區，但只有一人能進入危險區，而且這個人用完危險區後立刻離開，讓下一個人可以使用。

**真相揭曉**

多年後，當這套磨坊系統被寫入村史時，歷史學家阿萊總結出了三個關鍵教訓：

**第一個教訓：「可重入」的磨坊**
磨坊的操作分為「純查詢」和「狀態修改」。查詢操作（如「今天的轉速是多少」）可以被任意多個人同時執行，因為它們只是讀取狀態，不改變狀態。這就像程式設計中的「純函式」——相同輸入永遠產生相同輸出，不會造成任何副作用。

**第二個教訓：「不可重入」的危險**
任何改變齒輪狀態的操作，都必須在開始之前確保自己「獨占」了危險區。如果一個操作正在修改齒輪位置，另一個操作同時也在修改，就會發生災難性的後果——就像老陳的木頭被阿花的力道推偏一樣。這就是「race condition」（競態條件）。

**第三個教訓：「順序訪問」的解決方案**
有三種辦法可以避免競態條件：
1. **互斥鎖（操作鎖）**：同一時間只有一人能修改危險區。這是最直觀的方案。
2. **無鎖設計（獨立齒輪組）**：一開始阿花做的——把需要共享的部分徹底分開，每個人有自己的齒輪組。
3. **原子操作（精密聯動齒輪）**：用一種特殊設計的齒輪，讓「啟動並調整轉速」這兩個動作在機械上永遠是一個不可分割的原子操作，外部無法在兩個動作之間介入。

工程師最後說了一句話，被刻在磨坊的石碑上：

> 「**讓一個系統真正安全的，不是禁止所有人使用它，而是讓所有會產生衝突的操作，都必須經過一個共同的門——操作鎖。任何人在這個門裡的時候，其他人都必須在外面等。**」

## 概念解釋：執行緒安全與可重入性在系統函式庫中的實現

### 什麼是執行緒安全（Thread-Safety）？

一個函式被稱為「執行緒安全」，是指**多個執行緒同時調用它時，永遠能產生正確的結果**。反之，如果多執行緒同時調用會導致錯誤的結果、資料損壞或未定義行為，這個函式就是「非執行緒安全」的。

在寓言中，「非執行緒安全」對應的是：當老陳和阿花同時操作磨坊時，由於齒輪是共用的，他們的操作會相互干擾（就像兩個執行緒同時修改共享變數）。

### 什麼是可重入性（Reentrancy）？

可重入（reentrant）是指：**一個函式可以在被執行的途中，被同一執行緒再次呼叫並正確完成**。

具體來說，如果一個函式正在執行時，發生了一個訊號（signal），作業系統在訊號處理常式（signal handler）中再次呼叫同一個函式，該函式仍然能正確執行完畢，則它是可重入的。

可重入函式的特點：
- 不使用任何靜態（全域）資料或不可重入的資源
- 不呼叫任何不可重入的函式
- 不依賴任何鎖（因為訊號處理常式可能執行在任何上下文中）

**可重入的函式必然是執行緒安全的，但執行緒安全的函式不一定可重入**（因為執行緒安全可以透過加鎖來實現，但加鎖的函式通常不可重入——如果已經持有一把鎖，再次呼叫會造成鎖死）。

### 執行緒安全問題的根本原因：共享狀態

執行緒安全問題的根本，在於**多個執行緒同時訪問同一個可變的共享狀態**。在寓言中，「齒輪組的狀態」就是那個共享狀態。如果多個人同時操作齒輪組（改變轉速、調整位置），而不透過某種同步機制，就會發生混亂。

### 常見的非執行緒安全模式

1. **靜態資料競爭**：多執行緒同時讀寫同一個全域變數或靜態變數。

```c
// 非執行緒安全：全域變數 count
int count = 0;
void increment() {
    count++;  // 不是原子操作——實際上是：讀取→加一→寫回，三步之間可能被另一執行緒插入
}
```

2. **使用鎖但鎖的範圍不正確**：
```c
// 看似安全，實則不安全
void increment() {
    lock(&mutex);
    int temp = count;  // 持有鎖，但讀取後到寫入前有時間窗口
    temp++;
    count = temp;       // 寫入時可能已被其他執行緒修改
    unlock(&mutex);
}
```

3. **呼叫非執行緒安全的函式**：例如 `strtok()` 使用靜態緩衝區儲存狀態，連續呼叫會相互覆蓋。

### 實現執行緒安全的方法

**1. 互斥鎖（Mutex）**

互斥鎖是最常見的同步機制。同一時間只有一個執行緒可以持有鎖，其他執行緒試圖取得鎖時會被阻塞。

```c
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

void increment() {
    pthread_mutex_lock(&mutex);
    count++;
    pthread_mutex_unlock(&mutex);
}
```

Linux 的 `pthread` 庫提供以下幾種鎖：
- `PTHREAD_MUTEX_NORMAL`：不死鎖檢測，可能造成死鎖。
- `PTHREAD_MUTEX_RECURSIVE`：允許同一執行緒多次取得鎖，需相同次數釋放。
- `PTHREAD_MUTEX_ERRORCHECK`：提供錯誤檢測，適用於除錯。

**2. 讀寫鎖（RWLock）**

當讀操作遠多於寫操作時，讀寫鎖能提供更好的並發度：**多個執行緒可以同時讀取，但寫入時需要獨占**。

```c
pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

// 讀取
void read_count() {
    pthread_rwlock_rdlock(&rwlock);
    printf("count = %d\n", count);
    pthread_rwlock_unlock(&rwlock);
}

// 寫入
void increment() {
    pthread_rwlock_wrlock(&rwlock);
    count++;
    pthread_rwlock_unlock(&rwlock);
}
```

**3. 原子操作（Atomic Operations）**

對於簡單的整數讀寫，可以使用 GCC 的內建原子操作，避免使用鎖。

```c
// 使用 __atomic_add_fetch，確保 count++ 是原子操作
__atomic_add_fetch(&count, 1, __ATOMIC_SEQ_CST);
```

C11 的 `<stdatomic.h>` 提供了跨平台的原子操作介面：

```c
#include <stdatomic.h>
atomic_int count = ATOMIC_VAR_INIT(0);
void increment() {
    atomic_fetch_add_explicit(&count, 1, memory_order_seq_cst);
}
```

**4. 執行緒區域儲存（Thread-Local Storage, TLS）**

將共享狀態改為執行緒區域儲存，每個執行緒有自己的副本，完全避免衝突。

```c
// GCC / Clang
__thread int per_thread_counter;

// C11 / POSIX
pthread_key_create(&key, NULL);
pthread_setspecific(key, malloc(sizeof(int)));
```

**5. 無鎖設計（Lock-Free Design）**

使用硬體提供的原子指令（如 CAS——compare-and-swap）實現無鎖資料結構，適合高性能場景。

### Linux 系統函式庫中的執行緒安全實作

glibc（GNU C Library）是 Linux 系統中最重要的 C 標準函式庫。glibc 為每個非執行緒安全的函式都提供了對應的執行緒安全版本：

| 非執行緒安全版本 | 執行緒安全版本 | 機制 |
|----------------|--------------|------|
| `strtok()` | `strtok_r()` | 使用呼叫者提供的上下文指標 |
| `rand()` | `rand_r()` | 使用呼叫者提供的狀態指標 |
| `getenv()` | 無直接替代 | 通常在程式啟動後一次性讀取，之後不再訪問 |
| `asctime()` | `asctime_r()` | 使用呼叫者提供的緩衝區 |
| `gmtime()` | `gmtime_r()` | 使用呼叫者提供的結構體指標 |

這些 `_r` 後綴的版本（reentrant version，可重入版本）接受額外的指標參數，讓呼叫者提供自己管理的緩衝區或狀態指標，從而避免使用靜態共享狀態。

### errno 的執行緒安全問題

`errno` 是一個經典的非執行緒安全案例。在早期 Unix 系統中，`errno` 是全域變數。但在多執行緒環境下，多個執行緒同時呼叫系統呼叫並檢查 `errno`，會發生衝突。

Linux 採用的解決方案：**`errno` 在多執行緒程式裡實際上是每執行緒區域的（thread-local）**。glibc 將其實作為 TLS，具體由 `<errno.h>` 頭檔案中定義為 `__thread int errno`，每個執行緒有自己的 `errno` 副本。

### 可重入與訊號安全（Async-Signal-Safety）

一個函式如果同時滿足：
- 可重入
- 不呼叫任何非可重入函式

那它就是「async-signal-safe」的，可以在訊號處理常式中安全呼叫。

Linux 的 `signal-safety(7)` 手冊頁列出了所有可在訊號處理常式中安全呼叫的函式（約有兩百多個）。不在這個列表中的函式（如 `printf`、`malloc`、`free`）**不能在訊號處理常式中使用**，否則會造成死鎖或未定義行為。

### 總結

執行緒安全與可重入性，是 Linux 系統函式庫設計中一對相互交織但又有所不同的核心概念：

- **可重入性**描述的是一個函式在**被中斷後再次呼叫**時的行為——它是一個函式本身的屬性，不依賴外部同步機制。
- **執行緒安全**描述的是一個函式在**多執行緒並發環境**下的行為——它需要透過同步機制（鎖、原子操作、TLS）來保證。

在寓言的磨坊中，「操作鎖」對應的是互斥鎖；「獨立齒輪組」對應的是 TLS；「原子操作」對應的是不可分割的機械動作。**系統函式庫的設計者必須仔細區分哪些函式需要加鎖、哪些可以設計成無鎖可重入的，以確保整個系統在多執行緒環境下既高效又安全。**
